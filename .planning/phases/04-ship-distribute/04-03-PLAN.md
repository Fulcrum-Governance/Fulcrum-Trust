<task type="checkpoint:human-action" gate="blocking">
  <name>Checkpoint: Configure OIDC and push tag</name>
  <instructions>
This checkpoint requires two manual actions. Complete them in order.

**Action 1: Configure PyPI Trusted Publishing (if not already done)**

TestPyPI (do this first):
1. Go to https://test.pypi.org/manage/account/publishing/
2. If you don't have a TestPyPI account, register at https://test.pypi.org/account/register/ first
3. Fill in the "Add a new pending publisher" form:
   - PyPI project name: `fulcrum-trust`
   - Owner: `Fulcrum-Governance`
   - Repository name: `fulcrum-trust`
   - Workflow name: `publish.yml`
   - Environment name: `testpypi`
4. Click "Add"

Real PyPI:
1. Go to https://pypi.org/manage/account/publishing/
2. Fill in the same form with Environment name: `pypi`
3. Click "Add"

GitHub Environments (recommended):
1. Go to repo Settings → Environments
2. Create environment named `testpypi`
3. Create environment named `pypi` (optionally add a "Required reviewers" protection rule — this will pause before the real PyPI publish and wait for your manual approval)

**Action 2: Push the v0.1.0 tag**

Run these commands from your terminal (not Claude's shell — you need your git credentials):

```bash
# Create annotated tag
git tag -a v0.1.0 -m "v0.1.0 — initial release"

# Push the tag — this triggers the publish.yml workflow
git push origin v0.1.0
```

**After pushing the tag:**
1. Go to https://github.com/Fulcrum-Governance/fulcrum-trust/actions
2. Watch the "Publish" workflow run
3. The workflow has three stages: Build → Publish to TestPyPI → Publish to PyPI
4. If you added a manual approval gate on the `pypi` environment, approve it after TestPyPI succeeds
5. Wait for all three jobs to show green checkmarks

**If the workflow fails:**
- Build fails: check the build logs for metadata errors
- publish-testpypi fails with "No pending publisher": the OIDC configuration in step 1 wasn't saved — redo it
- publish-testpypi fails with name conflict: another user registered `fulcrum-trust` — contact PyPI support
- publish-pypi fails: check whether testpypi succeeded; if OIDC is the issue, verify environment names match exactly

Once all workflow jobs are green, type: "published"
  </instructions>
  <resume-signal>Type "published" when all three workflow jobs are green (Build, Publish to TestPyPI, Publish to PyPI)</resume-signal>
</task>

After the user confirms the workflow is green, run post-publish verification and create the GitHub release.

**Step 1: Verify TestPyPI install** (the workflow already published to TestPyPI; verify it works)
```bash
# Create a temporary clean virtualenv
python -m venv /tmp/testpypi-verify
/tmp/testpypi-verify/bin/pip install \
  --index-url https://test.pypi.org/simple/ \
  --extra-index-url https://pypi.org/simple/ \
  fulcrum-trust==0.1.0

# Verify import works
/tmp/testpypi-verify/bin/python -c "from fulcrum_trust import TrustManager; print('OK')"

# Clean up
rm -rf /tmp/testpypi-verify
```

**Step 2: Verify real PyPI install**
```bash
python -m venv /tmp/pypi-verify
/tmp/pypi-verify/bin/pip install fulcrum-trust==0.1.0
/tmp/pypi-verify/bin/python -c "from fulcrum_trust import TrustManager; print('OK')"
rm -rf /tmp/pypi-verify
```

Both must print "OK". If real PyPI install fails with "not found", wait 2-3 minutes for PyPI CDN propagation and retry.

**Step 3: Create GitHub release**

Extract the v0.1.0 section from CHANGELOG.md to use as release notes. The section runs from `## [0.1.0]` through the line before `[Unreleased]:`.

```bash
# Create the release (dist/ files must still exist from Task 1)
gh release create v0.1.0 \
  --title "v0.1.0 — Initial Release" \
  --notes "$(sed -n '/^## \[0\.1\.0\]/,/^\[Unreleased\]/{ /^\[Unreleased\]/d; p }' CHANGELOG.md)" \
  --latest \
  dist/fulcrum_trust-0.1.0-py3-none-any.whl \
  dist/fulcrum_trust-0.1.0.tar.gz
```

If the sed command is finicky, write the release notes to a temp file first:
```bash
# Alternative: write notes to file, then use --notes-file
grep -A 50 "## \[0\.1\.0\]" CHANGELOG.md | grep -B 50 "^\[Unreleased\]" | head -n -1 > /tmp/release-notes.md
gh release create v0.1.0 \
  --title "v0.1.0 — Initial Release" \
  --notes-file /tmp/release-notes.md \
  --latest \
  dist/fulcrum_trust-0.1.0-py3-none-any.whl \
  dist/fulcrum_trust-0.1.0.tar.gz
rm /tmp/release-notes.md
```

**Step 4: Confirm release**
```bash
gh release view v0.1.0
```

Output should show: tag v0.1.0, title "v0.1.0 — Initial Release", body with Added section from CHANGELOG, two assets attached (.whl and .tar.gz).
